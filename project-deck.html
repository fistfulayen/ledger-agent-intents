Ledger Agent Payments USDC Hackathon · Feb 2026

Agents propose.
Humans sign with hardware.

A secure bridge between AI agents and blockchain transactions: agents draft and propose payments, but only you approve and sign—on Ledger.

The promise
  • AI handles the work: discovery, negotiation, scheduling, bookkeeping.
  • You keep the keys: trusted display + physical confirmation.
  • USDC becomes the unit of account for the agent economy.
The slogan
"Your agent has root access. Your keys don't."

This deck is a product story first, then a technical deep dive at the end.

Open with the simplest framing: we’re building the “permission layer” for agent spending. The key is: agent can suggest anything; user can approve or reject on hardware.
The moment

AI agents are ready to act.
Money isn’t ready for agents.

Agents can already
  • Find vendors, compare prices, and negotiate.
  • Extract invoices from emails and calculate totals.
  • Call APIs, run jobs, and ship digital deliverables.
But payment is the cliff
  • Private keys + autonomy is a catastrophic combo.
  • Prompt injection or compromised tools become “empty the wallet”.
  • And humans need an audit trail, not a vibe.
Keep it concrete: “agents are already operating your browser and email”. The missing piece is safe money movement.
The problem

Agents need to spend.
But agents can’t hold keys.

If an agent can sign transactions, then a single failure mode (prompt injection, stolen token, malicious plugin) becomes irreversible loss.

What breaks today
  • “Approve this” dialogs without context.
  • Opaque hex payloads; users can’t reason about risk.
  • Automation without accountability: no durable trail.
What users need
  • Trusted display: amount, recipient, chain, reason.
  • Intent mediation: propose now, sign later.
  • Controls: approve / reject, with a recorded history.
The core is separation of duties: agent is the “planner”; Ledger is the “executor”.
The solution Intent Queue + Hardware Signing

A queue of intents.
A human-grade signing gate.

Agents create structured intents (who/what/why). Humans review the same structure and sign with Ledger when it’s correct.

This is the one-line product: Intent Queue + Ledger signing. Emphasize “propose now, sign later”.
The primitive

An Intent is a payment request with context.

Not “sign this hex”. An intent is a typed payload with clear fields: amount, token, recipient, chain, memo, urgency—and optional x402 metadata for API payments.

Minimal fields
  • Who: agentId / agentName, userId
  • What: token, amount, recipient, chainId
  • Why: memo, category, merchant
  • How urgent: urgency + expiration
Why it matters
  • Human review becomes fast and reliable.
  • Ledger signing becomes “clear signing”.
  • We can add rules later without changing the UX model.
Set up the “policy layer”: once intents are structured, you can do risk scoring, limits, multi-approval, etc.
The demo

A one-minute story.

User
  • “Clouseau, pay 50 USDC to Seth for the podcast intro music.”
  • Opens the app, sees a pending intent with full context.
  • Approves & signs on Ledger.

Technical API surface

A tiny REST API designed for agents.

Endpoints
GET /api/health POST /api/intents GET /api/intents/:id GET /api/users/:userId/intents?status=pending&limit=50 PATCH /api/intents/:id/status

Implemented as Vercel functions in apps/web/api.

Status lifecycle
  • pending → created by agent
  • approved / rejected → human review
  • signed → signature present (tx hash optional)
  • confirmed → chain confirmation
  • failed / expired → terminal outcomes
The surface area is intentionally small. The “status history” gives us auditability and product analytics (intent-to-sign time).
Technical Data model

Intents are rows + a status history timeline.

Tables (conceptually)
  • intents: `id`, `user_id`, `agent_id`, `agent_name`, `details`, `urgency`, timestamps
  • intent_status_history: `intent_id`, `status`, `timestamp`, `note`

In `apps/web/api/_lib/intentsRepo.ts` we also persist optional x402 proof data by embedding it into the JSON `details` blob.

Why status history matters
  • Audit trail (who did what, when).
  • UX: show progress and context for decisions.
  • Analytics: compute Intent-to-Sign time and drop-off.
Emphasize: it’s not “just a queue”. The history is the compliance / accountability layer.
Technical Signing

Ledger integration via an EIP-6963 provider.

What the web app does
  • Discovers a Ledger provider via EIP-6963 announcements.
  • Connects accounts (`eth_requestAccounts`).
  • Signs & sends tx (`eth_sendTransaction`) or typed data (`eth_signTypedData_v4`).
Why this is the right seam
  • The signing UI remains trusted (Ledger-controlled).
  • We can upgrade encoding/clear signing without changing the intent model.
  • Supports “on-chain tx hash” and “off-chain authorization signature” (x402-style).
We deliberately keep chain interactions behind the Ledger provider: the UI orchestrates, the hardware confirms.

Technical Security model

Threat model: assume the agent is compromised.

What an attacker can do
  • Create malicious intents (wrong recipient, wrong chain, wrong amount).
  • Spam intents to induce fatigue.
  • Try to remove context (“just sign this”).
What they can’t do
  • Sign without the Ledger device + human confirmation.
  • Bypass the trusted signing UI.
  • Erase the audit trail (status history) without server access.

Next hardening steps (not required for the hackathon): rules/limits, rate limiting, signature requirements for agents, and “high-risk” warnings.

The key is that compromise of the agent is survivable. Human approval is the safety boundary.
